What is Sonar?

Sonar is an advanced open source service and server monitoring tool designed to be extremely modular and light (both in installation size and in resources consumed, while still providing a framework for modules).

Sonar is intended to be run on a separate server. Preferably, even in a separate datacenter, so that datacenter link failures or targeted DDoSes don't affect the reliability of the monitoring platform. The cheapest VPS should be able to handle Sonar's infrastructure without much hassle.

Why? Isn't this similar to XXX, YYY and ZZZ?

While similar software exists, there is no major software implementation that was built from the ground up to be modular. Most monitoring tools offer an embedded web server that is usually ugly and not designed to be extensively modified.

Sonar's Status Engine (the core logic) is designed to be run as a daemon or periodically (through Linux's cron, for example), so that it doesn't constantly consume system resources.

Sonar's goal is also to be really light on resources, meaning that the actual system load will be the input and output modules and not the daemon itself, making it that much easier to optimize the infrastructure to suit low power machines.

Non-exahustive list of service and server monitoring software:

  • htop
  • Nagios
  • Zabbix
  • Munin
  • Cacti
  • Icinga
  • SmokePing

Why the silly name?

The idea behind Sonar is to have a central service (Status Engine) that periodically polls whatever the input modules provide (services, servers, IoT devices, or basically almost anything with a network connection), it's designed to do periodic and sequential queries (like a ship's Sonar) to each of the elements provided by input modules, so it works very similar to actual Sonar systems, since it doesn't need a service to be running on the target host, while it can still exahustively scan it.
